# 需求描述

构建一个 Skill 评测系统，支持单 Skill 评测与双 Skill 对比评测。对比评测要求：

1. 两个 Skill 并发执行，减少时间变量影响。
2. 两个 Skill 运行环境完全隔离，避免相互干扰。
3. 评测过程可追踪、可复现、可回放。

# 技术方案

## 总体架构

系统拆分为以下模块：

1. Gateway 服务：接收评测请求、推送执行事件、返回评测结果。
2. Agent Runtime：负责 LLM 循环、工具调用调度、重试控制。
3. Tool Registry：统一管理可用工具的注册、查询与调用。
4. Skill Loader：加载 Skill 元数据与内容，封装为统一调用格式。
5. Evaluator：执行单评测/对比评测并产出报告。
6. Report Store：保存 JSON 报告和执行产物。
7. Sandbox（Docker）：为每个评测任务提供独立执行环境。

## LLM Provider 策略

1. 当前阶段仅支持单一固定 LLM Provider（不做多 Provider 适配）。
2. 统一封装 Provider 调用接口，后续可平滑扩展多 Provider。
3. 记录每轮调用的 token 使用、耗时、请求和响应。

## Tool 设计与注册机制

### Tool 接口风格

工具统一采用 `get_weather` 当前实现的风格：

1. 每个工具提供标准参数定义（JSON Schema）。
2. 每个工具实现统一调用入口（入参、上下文、返回值、错误）。
3. 工具名称全局唯一，便于模型函数调用。

### Tool Registry

新增统一 Tool 注册中心，包含：

1. `Register(tool)`：注册工具（启动时完成）。
2. `Get(name)`：按名称获取工具实现。
3. `List()`：列出全部可用工具及参数描述。
4. `BuildToolParams()`：生成 LLM 可消费的 tools 列表。

### 默认工具集

1. 文件系统工具（filesystem，按统一接口风格改造）。
2. Shell 执行工具。
3. Python 执行工具（建议通过容器内执行）。
4. Use Skill 工具（触发 Skill 调用链路）。

## Skill 加载与调用

1. 支持从 `SKILL.md` 加载元信息（name、description、version）。
2. 支持加载 Skill 目录下资源文件并建立索引。
3. 支持将 Skill 封装为统一 API 调用格式，供 Agent Runtime 使用。
4. Skill 调用过程记录到评测日志（输入、输出、耗时、状态）。

## 评测用例管理

1. 用例以 JSON 保存，支持手工编辑与批量导入。
2. 用例包含：输入、期望目标、约束条件、评测标签。
3. 支持按场景分组，作为评测外层循环单位。
4. 每组用例支持随机顺序与固定顺序两种执行模式。

## 智能体循环（Agent Loop）

单个任务按如下循环执行：

1. LLM 生成回复或工具调用请求。
2. Runtime 解析工具调用，路由到 Tool Registry 执行。
3. 工具结果回填消息上下文，进入下一轮 LLM 调用。
4. 满足结束条件则停止：任务完成 / 达到最大轮次 / 发生不可恢复错误。

### 失败重试策略

1. LLM 调用失败：自动重试最多 3 次。
2. 工具执行失败：自动重试最多 3 次。
3. 达到 3 次仍失败：该任务标记为失败并进入报告。
4. 每次重试需记录失败原因、重试间隔和最终状态。

## Gateway 服务

1. 接收单评测和对比评测请求。
2. 支持流式推送执行事件（轮次状态、工具调用、错误、完成事件）。
3. 提供评测结果查询接口（按任务 ID）。
4. 提供报告下载接口（JSON）。

## 评测执行模型

1. 外层循环：按用例组执行。
2. 内层循环：按组内用例逐条执行 Agent Loop。
3. 使用 goroutine 并发调度评测任务。

### 对比评测隔离方案（Docker）

1. 对比评测的每个 Skill 在独立 Docker 容器中运行。
2. 容器内提供隔离文件系统、Shell 执行环境、Python 执行环境。
3. 容器间网络和挂载按最小权限配置，防止相互污染。
4. 每次评测完成后销毁容器，确保下一次评测环境干净。

## 评测判分机制

采用“AI 判断 + 人工确认”的双阶段机制：

1. AI 初判：基于用例目标与执行轨迹进行自动评分与结论生成。
2. 人工复核：对 AI 结论进行确认、修正或驳回。
3. 最终结果以人工确认为准，并保留 AI 初判内容用于追踪。

## 评测维度

评测报告至少包含以下维度：

1. 任务成功率：是否完成目标、失败类型分布。
2. 结果质量：输出正确性、完整性、可用性。
3. 工具使用质量：调用是否合理、是否存在无效调用。
4. 执行效率：总耗时、平均轮次、单轮耗时。
5. 资源消耗：token 消耗（输入/输出/总计）。
6. 稳定性：重试次数、异常率、超时率。
7. 产物质量：生成文件/脚本/结构化结果的可用性。

## 评测报告格式（JSON）

报告需支持通用分析与回放，字段建议包括：

1. 基本信息：任务 ID、评测类型、Skill 信息、开始/结束时间。
2. 用例信息：用例组、用例输入、期望目标、标签。
3. 过程明细：每轮 LLM 请求/响应、工具调用入参与返回值。
4. 失败与重试：错误栈、重试次数、最终失败原因。
5. 指标统计：token、耗时、轮次、成功率、异常率。
6. 判分结果：AI 初判结论、人工确认结论、最终评分。
7. 产物清单：生成的文件、路径、摘要和可访问地址（如有）。

## 里程碑建议

1. M1：打通单 Skill 评测闭环（Agent Loop + Tool Registry + JSON 报告）。
2. M2：接入 Docker 隔离并支持双 Skill 对比评测。
3. M3：完善 AI 判分与人工确认工作流。
4. M4：补齐可观测性和前端事件流展示能力。
